# <a name="-10-الحوسبة-السحابية-cloud-computing"></a>☁️ الفصل 10: الحوسبة السحابية (Cloud Computing)

## 1. ما هو التخصص؟ (What is it?)

الحوسبة السحابية هي توفير الموارد الحاسوبية (مثل الخوادم، التخزين، قواعد البيانات، الشبكات، البرمجيات) عند الطلب عبر الإنترنت، مع
نموذج تسعير يعتمد على الدفع مقابل الاستخدام (Pay-as-you-go).

ببساطة، بدلاً من شراء وامتلاك وصيانة مراكز بيانات وخوادم فعلية، يمكنك الوصول إلى هذه الخدمات التكنولوجية من مزود سحابي.

القياس الأفضل: بدلاً من بناء محطة كهرباء خاصة بك، أنت تشترك في شركة الكهرباء العامة وتدفع فقط مقابل ما تستهلكه. السحابة هي "شركة
الكهرباء" لعالم الحوسبة. أشهر المزودين هم AWS, Azure, و GCP.

## 2. ماذا يفعل المتخصص فيه؟ (What does the specialist do?)

مهندس السحابة (Cloud Engineer) هو الذي يصمم، يبني، ينشر، ويدير التطبيقات والبنية التحتية على المنصات السحابية. مهامه تشمل:

* تصميم الحلول السحابية: تصميم بنية تحتية آمنة، قابلة للتوسع، ومتاحة دائماً على السحابة.
* ترحيل الأنظمة (Migration): مساعدة الشركات على نقل تطبيقاتها وبنيتها التحتية الحالية من الخوادم المحلية (On-premise) إلى السحابة.
* إدارة البنية التحتية ككود (IaC): كتابة كود لإنشاء وإدارة الموارد السحابية بشكل آلي.
* تحسين التكاليف (Cost Optimization): مراقبة وتحليل الإنفاق على السحابة وإيجاد طرق لتقليل التكاليف.
* ضمان الأمان والامتثال: تطبيق أفضل الممارسات الأمنية للتأكد من أن البيئة السحابية محمية.

## 3. الفروع الداخلية للتخصص (Sub-domains)

* هندسة الحلول السحابية (Cloud Architecture): يركز على التصميم عالي المستوى للبنية السحابية.
* إدارة السحابة (Cloud Administration): يركز على العمليات اليومية لإدارة البيئة السحابية.
* أمان السحابة (Cloud Security): متخصص في حماية البنية التحتية والبيانات على السحابة.
* شبكات السحابة (Cloud Networking): متخصص في تصميم وإدارة الشبكات الافتراضية داخل السحابة.
* FinOps (إدارة الشؤون المالية للسحابة): تخصص جديد يركز على إدارة وتحسين تكاليف السحابة.

## 4. قائمة موسعة بالمفاهيم الأساسية (Expanded Key Concepts)

* نماذج الخدمة ("as a Service"):
  * `IaaS (Infrastructure as a Service)`: أنت تستأجر البنية التحتية الأساسية (خوادم افتراضية، تخزين). أنت تدير نظام التشغيل وكل شيء
       فوقه. (مثال: AWS EC2).
  * `PaaS (Platform as a Service)`: المزود يدير لك المنصة (نظام التشغيل، بيئة التشغيل). أنت فقط تنشر الكود الخاص بك. (مثال: Heroku).
  * `SaaS (Software as a Service)`: أنت تستخدم برنامجاً كاملاً وجاهزاً عبر الإنترنت. (مثال: Gmail).
* نماذج النشر (Deployment Models):
  * `Public Cloud`: الموارد مملوكة لمزود الخدمة ومتاحة للعامة عبر الإنترنت.
  * `Private Cloud`: الموارد مستخدمة حصرياً من قبل شركة واحدة.
  * `Hybrid Cloud`: مزيج بين السحابة العامة والخاصة.
* المبادئ الأساسية للسحابة:
  * `Scalability & Elasticity`: قابلية التوسع (زيادة الموارد) والمرونة (زيادة الموارد تلقائياً بناءً على الطلب).
  * `High Availability & Fault Tolerance`: تصميم أنظمة تظل تعمل حتى لو فشلت بعض مكوناتها، غالباً عن طريق توزيعها على مناطق جغرافية
       مختلفة.
* `Serverless Computing` (الحوسبة بدون خوادم): تطور للـ PaaS حيث يدير المزود كل شيء. أنت تكتب الكود كـ "دوال" (Functions) تعمل فقط عند
    استدعائها، وتدفع فقط مقابل وقت التشغيل بالمللي ثانية. (مثال: AWS Lambda).

## 5. قائمة موسعة بالأدوات والتقنيات (Expanded Tools & Technologies)

* مزودو السحابة: AWS (Amazon Web Services), Microsoft Azure, GCP (Google Cloud Platform).
* خدمات الحوسبة: AWS EC2, Azure VMs (IaaS); AWS Lambda, Azure Functions (Serverless).
* خدمات التخزين: AWS S3, Azure Blob Storage (تخزين الكائنات); AWS EBS (تخزين الكتل).
* خدمات قواعد البيانات: Amazon RDS (SQL مُدارة); Amazon DynamoDB (NoSQL مُدارة).
* خدمات الشبكات: Amazon VPC (شبكات افتراضية); AWS ELB (موازنات الحمل).
* البنية التحتية ككود (IaC): Terraform, AWS CloudFormation.
* الحاويات: Docker, والخدمات المُدارة لـ Kubernetes مثل Amazon EKS, Azure AKS, Google GKE.

## 6. سيناريو عملي معمق (In-Depth Workflow)

المشروع: ترحيل تطبيق ويب من خادم محلي إلى بنية قابلة للتوسع ومتاحة دائماً على AWS.

 1. التقييم والتصميم: يحلل مهندس السحابة التطبيق الحالي. يقرر تشغيل خوادم الويب على عدة EC2 instances (خوادم افتراضية) لتحقيق التوافر،
    وترحيل قاعدة البيانات إلى Amazon RDS (خدمة قواعد بيانات مُدارة).
 2. كتابة كود البنية التحتية (IaC): بدلاً من إنشاء الموارد يدوياً، يكتب المهندس سكربت Terraform يصف البنية التحتية المطلوبة:
     * شبكة افتراضية خاصة (VPC).
     * مجموعة توسع تلقائي (Auto Scaling Group) لزيادة عدد خوادم EC2 إذا زاد استخدام المعالج عن 70%.
     * موازن حمل (Load Balancer) لتوزيع الزوار على الخوادم.
     * قاعدة بيانات RDS مع نسخة احتياطية في منطقة توافر (Availability Zone) مختلفة.
 3. النشر: يقوم المهندس بتشغيل terraform apply. يقوم Terraform بالتواصل مع AWS وبناء كل هذه البنية التحتية تلقائياً.
 4. ترحيل البيانات: يقوم المهندس بتصدير البيانات من قاعدة البيانات المحلية واستيرادها إلى قاعدة بيانات RDS الجديدة.
 5. نشر التطبيق: يقوم فريق DevOps بتحديث خط أنابيب CI/CD لنشر كود التطبيق على خوادم EC2 الجديدة.
 6. التحويل: بعد الاختبار، يتم تحديث سجل DNS الخاص باسم النطاق ليشير إلى عنوان موازن الحمل الجديد. يبدأ الزوار الآن بالوصول إلى التطبيق
    على السحابة.
 7. المراقبة والتحسين: يستخدم المهندس Amazon CloudWatch لمراقبة أداء البنية التحتية. قد يلاحظ ضغطاً على قاعدة البيانات ويقرر ترقيتها، أو
    قد يقوم بإعداد تنبيهات لتتبع التكاليف.

## 7. الأدوار الوظيفية الشائعة (Common Job Roles)

* Cloud Engineer: مهندس سحابة.
* Cloud Architect: معماري سحابة.
* Cloud Support Engineer: مهندس دعم سحابي.
* Cloud Security Engineer: مهندس أمان سحابي.
* FinOps Specialist: أخصائي إدارة مالية للسحابة.
